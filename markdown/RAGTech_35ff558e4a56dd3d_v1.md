第一章 什麼是 RAG？
RAG（Retrieval-Augmented
Generation，檢索增強生成）是一種結合「資訊檢索」與「大型語言模型生成」的 AI 架構。它讓 LLM
在回答問題前，先從外部知識庫中找到相關資料，再將資料整合進提示詞（Prompt），從而生成更準確
、更有來源根據的答案。
 傳統 LLM vs. RAG 比較
比較項目 傳統 LLM RAG 架構
知識來源 訓練資料（靜態） 外部知識庫（動態）
知識更新 需重新訓練 即時更新文件即可
幻覺問題 較高機率 大幅降低
可追溯性 無法追蹤來源 附帶文件來源引用
適用情境 通用對話 企業知識、專業領域
 RAG 核心優勢
即時更新 可追溯性 成本效益 資料隱私

RAG 系統包含兩大核心階段：離線的「索引建立」與線上的「檢索生成」。下圖展示完整的資料流向：

 流程步驟詳解
① 文件前處理 將 PDF、Word、網頁等原始文件切割成適當大小的文字區塊（Chunk），常見策
略包含固定長度分割、語意分割與遞迴分割。
② 向量嵌入 使用 Embedding 模型（如 text-embedding-3-small）將每個 Chunk
轉換為高維度向量，捕捉語意資訊。
③ 向量儲存 將向量存入向量資料庫（ChromaDB、Pinecone、Qdrant
等），建立可快速詢的索引結構。
④ 問題檢索 使用者提問後，問題同樣被嵌入為向量，再以餘弦相似度或 BM25
混合檢索，找出最相關的 Top-K 個 Chunk。
⑤ 提示詞組合 將檢索到的 Chunk 與原始問題組合成增強提示詞，指示 LLM
僅根據提供資料回答。
⑥ 生成回答 LLM
根據增強提示詞生成最終回答，並可附上文件來源引用，提升答案可追溯性。

第四章 文件切割（Chunking）策略
切割策略直接影響 RAG 的檢索品質。Chunk 太大會稀釋關鍵資訊；Chunk
太小則缺乏語境。需依據文件類型與應用場景選擇最適合的策略。
依字元或 Token 數切割，簡單快速。 依段落、標題或語意邊界切割，保留完整語境，
固定長度切割 語意切割 建議設定：512 tokens，重疊 50 tokens。 適合結構化文件。
Fixed-size Chunking Semantic Chunking
先以段落切割，過長再細分，LangChain 保留文件層級（章節→段落→句子），適合長文
遞迴切割 階層式切割 預設策略，兼顧效率與品質。 件的多層次檢索。
Recursive Chunking Hierarchical Chunking
 實踐建議 一般文件建議從 512 tokens + 50 tokens 重疊開始，再根據測試結果調整。RAG
評估指標可使用 RAGAS 框架（Faithfulness、Answer Relevancy、Context Recall）量化品質。

第六章 導入最佳實踐
01 文件品質優先
進出（GIGO）原則同樣適用於
RAG。確保輸入文件結構清晰、容準確、版本最新，是系統品質的根本保障。
02 混合檢索策略
單純向量檢索對關鍵字的敏感度較低，建議結合 BM25 關鍵字檢索與向量語意檢索，採用
RRF（倒數排名融合）演算法合併結果，提升召回率。
03 重排序（Reranking）
初步檢索後，使用 Cross-Encoder 重排序模型（如 bge-reranker）對候選 Chunk
進行精確相關性排序，僅將 Top-3 送入 LLM，節省 Context 空間。
04 評估與監控
上線後持續使用 RAGAS 等框架評估系統品質，追蹤 Faithfulness（忠實度）、Answer
Relevancy（相關性）、Context Precision（精確率）等指標。
05 Prompt 工程
明確指示 LLM「僅根據提供的參考資料回答」、「若無相關資料請如實告知」，可有效減少
幻覺並提升回答可信度。

都提供了一條低成本、高效益的 AI 知識管理路徑。

證（PoC）開始，逐步擴展至生環境，並持續根據評估指標優化各個環節。
